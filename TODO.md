这是一份为你量身定制的 **Go + K3s 即时通讯（IM）系统从 0 到 1 的项目设计路线图**。

这份路线图遵循 **“敏捷迭代、小步快跑”** 的原则，将庞大的 IM 系统拆解为 5 个可执行的阶段。每个阶段都有明确的**核心目标**、**技术重点**和**交付成果**。作为新手，请严格按顺序执行，不要试图一次性完成所有功能。

---

### 🗺️ 项目总览：架构演进路线
*   **阶段一**：单机版回声室（验证网络模型与容器化）
*   **阶段二**：基础聊天室（实现用户管理与单聊/群聊逻辑）
*   **阶段三**：数据持久化与状态管理（引入数据库与缓存）
*   **阶段四**：分布式集群架构（解决多节点通信问题，核心难点）
*   **阶段五**：生产级加固（监控、日志、高可用部署）

---

### 🚀 阶段一：最小可行性产品 (MVP) - "Hello WebSocket"
**目标**：跑通 "客户端 <-> Go 服务 <-> 浏览器" 的全链路，并成功在 K3s 中运行一个无状态的 Docker 容器。

#### 1. 核心任务
*   **协议实现**：使用 `gorilla/websocket` 实现基础的 WebSocket 握手、心跳检测（Ping/Pong）、断线重连机制。
*   **并发模型**：实现基于 `Goroutine` + `Channel` 的单进程消息广播模型（Hub 模式）。
*   **容器化**：编写多阶段 `Dockerfile`，将镜像体积控制在 20MB 以内。
*   **K3s 部署**：编写 `Deployment` 和 `NodePort Service` YAML，将服务暴露给局域网访问。

#### 2. 技术重点
*   理解 Go 的 `select` 语句在连接管理中的作用。
*   掌握 K3s 的 `kubectl` 基本命令 (`apply`, `get`, `logs`, `exec`)。
*   理解 Docker 镜像分层与多阶段构建。

#### 3. 交付成果
*   一个可运行的 Web 页面，打开两个标签页可以互相发送文本消息。
*   代码仓库初始化，包含完整的 CI/CD 基础结构（可选：简单的 Makefile）。
*   **里程碑**：在 K3s 集群中重启 Pod，聊天不崩溃（虽然会断开，但服务能自动拉起）。

---

### 🏗️ 阶段二：业务逻辑构建 - "真正的聊天"
**目标**：从“广播室”升级为“聊天室”，实现用户身份识别、私聊和群组消息路由。

#### 1. 核心任务
*   **用户体系**：设计简单的用户模型（UserID, Nickname），在 WebSocket 握手时通过 Query 参数或 Header 传递 Token 进行模拟认证。
*   **连接映射**：在内存中维护 `Map<UserID, Connection>`，实现精准的消息投递（单聊）。
*   **群组逻辑**：设计 `Map<GroupID, Set<UserID>>`，实现群组消息广播。
*   **离线消息（内存版）**：当用户未在线时，将消息暂存在内存队列中，用户上线后拉取（重启即丢失，仅做逻辑验证）。

#### 2. 技术重点
*   设计高效的数据结构来管理百万级连接（考虑读写锁 `RWMutex` 的使用）。
*   错误处理：处理非法消息格式、恶意连接、频繁重连。
*   日志规范：引入 `zap` 日志库，区分 Info/Error 级别。

#### 3. 交付成果
*   支持添加好友、创建群组。
*   支持一对一私聊，消息不会泄露给第三人。
*   支持群组聊天。
*   **里程碑**：在 K3s 中扩缩容（Scale）Pod 数量，观察内存中连接状态的变化（此时多节点间消息会不通，这是正常的，为下一阶段做铺垫）。

---

### 💾 阶段三：数据持久化与状态分离 - "记忆与缓存"
**目标**：引入中间件，解决数据丢失问题，并将“连接状态”与“业务数据”分离。

#### 1. 核心任务
*   **数据库集成**：
    *   选型：MySQL (存储用户信息、群组关系、历史消息)。
    *   实现：异步写入消息历史（避免阻塞主线程），设计消息表结构（`msg_id`, `sender`, `receiver`, `content`, `timestamp`）。
*   **缓存集成**：
    *   选型：Redis。
    *   用途 1：存储用户在线状态（`Online: UserID -> ServerIP`）。
    *   用途 2：存储用户未读消息计数。
    *   用途 3：作为分布式锁，防止并发写冲突。
*   **配置中心**：使用 `ConfigMap` 管理 K3s 中的配置文件，实现代码与配置分离。

#### 2. 技术重点
*   **异步 IO**：使用 Go 的 Worker Pool 模式处理数据库写入，防止高并发拖垮 DB。
*   **连接池管理**：优化 MySQL 和 Redis 的连接池参数。
*   **K3s 有状态服务**：使用 `Helm` 或 `Operator` 在 K3s 中部署高可用的 Redis 和 MySQL（或使用 `StatefulSet`）。

#### 3. 交付成果
*   重启服务器后，历史消息可从数据库加载。
*   用户上线时，能收到离线期间的消息。
*   能通过 Redis 查询到哪些用户当前在线。
*   **里程碑**：模拟数据库宕机，服务应具备降级能力（如暂时只发消息不存库，或返回友好提示），而不是直接 Crash。

### ☁️ 阶段四：分布式集群架构 - "打破单机限制" (核心难点)
**目标**：解决多节点（Multi-Pod）通信问题，实现真正的水平扩展。这是新手向专家跨越的关键一步。

#### 1. 核心任务
*   **问题分析**：当用户 A 连在 Pod 1，用户 B 连在 Pod 2 时，内存路由失效。
*   **解决方案：Redis Pub/Sub (发布订阅)**
    *   所有 Pod 订阅同一个全局频道（或按用户哈希分片）。
    *   Pod 1 收到 A 发给 B 的消息 -> 检查本地是否有 B 的连接 -> 若无 -> 发布消息到 Redis `channel:user_B`。
    *   Pod 2 监听到 `channel:user_B` -> 发现 B 在自己这里 -> 推送消息给 B。
*   **负载均衡**：配置 K3s 的 `Ingress` (如 Nginx Ingress)，实现 WebSocket 的长连接负载均衡（需开启 `sticky sessions` 粘滞会话，虽非必须但推荐）。
*   **服务发现**：利用 K8s 内部的 DNS 服务发现机制。

#### 2. 技术重点
*   **分布式一致性**：理解最终一致性在 IM 中的应用。
*   **消息去重**：防止网络抖动导致消息重复投递（引入 `MessageID` 机制）。
*   **优雅停机 (Graceful Shutdown)**：捕获 K8s 的 `SIGTERM` 信号，在 Pod 销毁前通知客户端重连，并清理 Redis 中的在线状态。

#### 3. 交付成果
*   在 K3s 中将副本数扩展到 3 个，任意两个用户无论连接到哪个 Pod，都能正常通信。
*   手动删除一个 Pod，用户无感知（或短暂重连后恢复），消息不丢失。
*   **里程碑**：压测。使用工具（如 `wsbench`）模拟 1000+ 并发连接，观察 CPU/内存水位及消息延迟。

### 🛡️ 阶段五：生产级加固与运维 - "稳如磐石"
**目标**：完善监控、安全和高可用策略，使项目达到可上线标准。

#### 1. 核心任务
*   **可观测性**：
    *   **指标监控**：集成 `Prometheus` + `Grafana`，监控指标包括：在线连接数、消息吞吐量 (QPS)、消息延迟、戈协程数量、内存使用率。
    *   **链路追踪**：(可选) 集成 `Jaeger` 或 `OpenTelemetry`，追踪消息从接入到落库的全流程。
*   **安全性**：
    *   启用 `WSS` (WebSocket over TLS)，在 K3s 中配置 `cert-manager` 自动签发证书。
    *   实现完善的鉴权中间件（JWT 校验）。
    *   防攻击：限制单 IP 连接数、消息频率限制（Rate Limiting）。
*   **CI/CD 流水线**：
    *   编写 GitHub Actions 或 GitLab CI 脚本：代码提交 -> 自动测试 -> 构建镜像 -> 推送仓库 -> 更新 K3s 部署。
*   **资源限制**：在 K8s YAML 中配置 `requests` 和 `limits`，防止单个 Pod 耗尽节点资源。

#### 2. 技术重点
*   K8s 的 `HPA` (Horizontal Pod Autoscaler)：根据 CPU 或自定义指标（如连接数）自动扩缩容。
*   网络安全策略 (`NetworkPolicy`)：限制 Pod 间的非必要访问。

#### 3. 交付成果
*   一套完整的监控大屏，实时展示 IM 系统健康度。
*   全自动化的发布流程。
*   支持 HTTPS/WSS 安全访问。
*   **最终里程碑**：系统在 K3s 集群中稳定运行 7x24 小时，经受住压力测试，具备自愈能力。

### 💡 给新手的特别提示
1.  **不要过度设计**：在阶段一和阶段二，不要纠结于微服务拆分，单体应用足够支撑数万并发。
2.  **重视日志**：在分布式环境下，日志是你唯一的眼睛。
3.  **K3s 是朋友**：遇到 K8s 问题，先用 `kubectl describe pod <pod-name>` 和 `kubectl logs <pod-name>` 查看原因，90% 的问题都能通过这两个命令解决。
4.  **参考开源**：遇到困难时，可以参考 `go-im-server` 或 `Tao` (GitHub 上的开源 IM 项目) 的源码思路，但不要直接复制，要自己手写以理解原理。

这份路线图就是你的导航仪。每完成一个阶段，你都会对后端架构和云原生技术有更深的理解。准备好开始第一阶段了吗？
