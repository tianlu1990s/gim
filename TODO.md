# GIM 项目开发路线图

本文档详细描述了 GIM 项目的开发阶段、各阶段目标及关键任务。

##  阶段一：原型与核心链路 (MVP)

**目标**: 跑通“客户端-网关-逻辑-存储-客户端”的最小闭环，实现基础的单聊功能。

- [ ] **基础设施搭建**
    - [ ] 初始化 Go Module 项目结构。
    - [ ] 编写 Docker Compose 文件，启动 MySQL, Redis。
    - [ ] 配置 Zap 日志库和 Viper 配置管理。
- [ ] **接入层开发 (Connect)**
    - [ ] 实现基于 `gorilla/websocket` 的 WebSocket 服务器。
    - [ ] 实现简单的 Ping/Pong 心跳机制。
    - [ ] 定义基础通信协议 (JSON 格式)。
- [ ] **逻辑层开发 (Logic)**
    - [ ] 实现用户登录鉴权接口 (生成 Token)。
    - [ ] 实现单聊消息的接收与简单回显。
- [ ] **存储层开发 (Storage)**
    - [ ] 设计用户表 (`user`) 和会话表 (`conversation`)。
    - [ ] 实现消息入库 (MySQL) 的基本 DAO 层。

##  阶段二：架构升级与消息可靠性

**目标**: 引入消息队列解耦，实现消息的持久化、ACK 机制和离线消息处理。

- [ ] **引入消息队列 (Kafka)**
    - [ ] 部署 Kafka 集群。
    - [ ] 接入层将上行消息异步写入 Kafka。
    - [ ] 逻辑层消费 Kafka 消息进行业务处理。
- [ ] **消息可靠性设计**
    - [ ] 实现应用层 ACK 机制 (发送方等待接收方确认)。
    - [ ] 设计消息序列号 (`seq_id`)，用于检测丢包和乱序。
- [ ] **离线消息支持**
    - [ ] 设计“未读消息表”或“消息游标”。
    - [ ] 实现用户上线后的拉取接口 (`/msg/sync`)，根据 `seq_id` 拉取历史消息。
- [ ] **引入 MongoDB**
    - [ ] 将消息内容从 MySQL 迁移至 MongoDB，以支持海量存储。

##  阶段三：群聊与复杂业务

**目标**: 支持群组聊天，优化群聊性能，完善用户关系链。

- [ ] **群组系统**
    - [ ] 数据库设计：群表、群成员表、群管理员表。
    - [ ] 实现群的创建、加入、退出、踢人接口。
- [ ] **群聊消息流转**
    - [ ] **写扩散实现**: 适用于 50 人以下小群。
    - [ ] **读扩散实现**: 适用于大群，设计“群时间线”集合。
- [ ] **好友关系**
    - [ ] 实现好友申请、通过、拉黑、删除功能。
    - [ ] 实现通讯录同步接口。

##  阶段四：分布式与高可用

**目标**: 支持多网关节点部署，实现水平扩展，解决路由问题。

- [ ] **服务发现与注册**
    - [ ] 引入 Etcd/Consul。
    - [ ] 网关启动时注册自身 IP，宕机时自动剔除。
- [ ] **路由服务 (Router)**
    - [ ] 实现全局路由表 (Redis Hash 结构: `uid -> gateway_ip`)。
    - [ ] 逻辑层通过 RPC 跨节点推送消息 (当用户连接在不同网关时)。
- [ ] **负载均衡**
    - [ ] 配置 Nginx 对 WebSocket 连接进行负载均衡。

##  阶段五：高级特性与优化

**目标**: 丰富消息类型，提升性能，增加监控。

- [ ] **富媒体消息**
    - [ ] 对接对象存储 (MinIO/OSS)，支持图片、文件上传下载。
    - [ ] 扩展消息协议支持自定义信令。
- [ ] **性能优化**
    - [ ] 使用 `sync.Pool` 减少内存分配。
    - [ ] 优化 Redis 批量操作。
- [ ] **监控与运维**
    - [ ] 接入 Prometheus + Grafana，监控连接数、消息吞吐量、延迟。
    - [ ] 接入 Jaeger 进行全链路追踪。
